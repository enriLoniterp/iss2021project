PMS

From the backlog, the PMS needs to processes car deposit and pick-up requests.
Following the single-responsability principle, the introduction of a new entity would allow the PMS to perform these tasks without interferring with the rest of the application. Transport trolley interaction occurs with the TrolleyController developed in the previous sprint.
Regarding the interaction with the client, as requested by the customer, it is necessary to generate an unique receipt (TOKENID) every time a client request a deposit which will be used during pick-up phase.
As per requirements, the TOKENID has to be representative of the SLOTNUM assigned to the parked car. An unique TOKENID can be generated following two strategies:
  • using current date and time + SLOTNUM;
  • random string long enough to avoid recurrences + SLOTNUM.
During the pick-up phase, DTFREE interval must be correctly defined according to parking requirements. In particular, if pick-up request is done near the parking, DTFREE should be relatively short, longer otherwise.
If all parking slot are engaged and the client makes an acceptIN request, PMS should reply with a fail message.

Another necessity is the managment of the parking status using a specific datastructure. In case of system crashes, storing the parking status in a local file make the system robust.

ParkServiceGUI

ParkServiceGUI is a user interface which allows clients interact with the parking system and has already been analyzed in Automated Car-Parking.html.

ParkServiceGUI - PMS interaction

The interaction between these two entites takes place through different messages pointed out here using the same communication logic already detailed (request/response and dispatches).

Given the distributed nature of the application and the presence of concurrency requests, we highlight some particular scenario:
  1. client requests the service but does not complete it.
  2. the application receives two concurrent requests: the system must be able to queue all the requests received in one or eventually more queues based on priorities and execute them in a sequential way (FIFO).
  3. After an acceptIN which still miss the relative CARENTER phase and another acceptIN request arrives, CARENTER phase can have some problem: in this scenario, to avoid any conflicts CARENTER must encode the information about the SLOTNUM priovously received.
A model of the two entites and their interaction is available here.

Project

Starting from what suggested in the analysis, we decided to introduce a new entity called parkClientService which processes clients' requests and is part of the PMS.
We decided to adopt the MVC pattern to develop the ParkServiceGUI, splitting it into two components (Model not needed in our case):
  • View: user interfaces.
  • Controller: receives requests and generates content view for the client.

ParkingState

For the sake of semplicity we represented parking state using a single resource which encodes the following variable:
  • Transport trolley state
  • Fan state
  • Temperature
  • indoor state
  • outdoor state
  • Parking slots state
This resource is a seriazable kotlin class which can be easily saved into a json file.

parkClientService


In order to accept, queue and process different requests in a non-blocking manner, the parkClientService is implemented as an actor which has multiple status to exploit different tasks. parkClientService logic is message-based so that its state changes are triggered by messages or events.
In addition, parkClientService needs to take care about the current state of the system.
The scheme on the side shows parkClientService behaviour.

parkClientService communication

parkClientService interacts with GuiAdapter, from which receives client's requests and to which sends tasks outcomes using a request/response logic. In order to send QAK understandable messages, GuiAdapter uses connQakTcp.kt, MsgUtil class and ApplMessageUtil class.
As said during the analysis phase, parkClientService interacts with trolleyController using the semantic described in Sprint2 to move the robot and fulfill client requests. Finally, parkClientService needs to stay updated about the current data sent by the sensors through ParkingState.

ParkServiceGUI

The Spring Framework will be used to implement the MCV pattern (SpringMVC and SpringBoot in particular) for its ability to boost productivity with the use of annotations, Dependency Injection and easily configurable web services. In addition, Spring is a light-weight container which guarantees a minimal overhead and an easy integration with current services.

GuiAdapter

This component is the SpringController which generates the right View for the client and fulfills his requests by interacting with parkClientService. We used annotations to let Spring container managing functionalities not linked to business-logic.

ParkServiceView

Multiple html files have been written to represents client views:
  • home.html
  • pickup.html
  • deposit.html
These HTML pages contains javascript code to make requests to the controller and to dynamically modify html DOM.
