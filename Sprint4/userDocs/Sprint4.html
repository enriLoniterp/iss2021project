Final task ISS-2021 Bologna: Automated Car-Parking - Sprint 4

Sprint 4 Backlog

Sprint 4 regards the develop of the PMS able to manage interaction with manager and care of system status:
  • PMS satisfies manager's start and stop trolley and fan requests: R4, R5.
  • PMS maintains an updated version of the system status: R4.
  • PMS recognizes critical OUTDOOR-area situation and generates an alarm for the manager: R4.

PMS

From the backlog, the PMS needs to process manager's requests and provide information about current system status.
As suggested in previous Sprint, the introduction of a new entity allows to process manager's tasks without affect the rest of the application. In particular, thanks to the Parking State described in the last Sprint, this Sprint will focus on how to properly send that data to the manager GUI. Since the data represent the current system status, every change must be notified as soon as possibile.
In addition, manager's task higly depends on park temperature and he should be properly notified when the temperatue exceeds TMAX threshold. In fact, the manager is able to start the trolley and turn off the fan if temperature is below TMAX and turn on the fan and start stop the trolley otherwise.
In order to avoid chances of human errors, fan management (turn on/turn off) should be done automatically by interacting with FanAdapter. Meanwhile Transport Trolley management is left exclusively to the manager himself and can be done using the TrolleyController.
Finally, PMS must be able to support some kind of timer which times car presence in the OUTDOOR AREA for more than DTFREE time.

ParkServiceStatusGUI

ParkServiceStatusGUI is a user interface which allows managers to interact with the parking system and has already been analyzed here.

ParkServiceGUI - PMS interaction

Interaction between these two entites takes place through different messages pointed out here using the same communication logic already detailed (request/response and dispatches). Data relative to the parking state can be exchanged using:
  • POLLING: requests are continuously sent by the GUI. This methodology affects client performance.
  • Open communication channel: PMS continuously send new data to the client side. This methodology affects server performance.
Both methods can be implemented using HTTP protocol or Websocket.
Alarm management can be treated in the same way, but the polling mode doesn't really fit this scenario as the alarm event occurs discontinuously and not periodically differently from the change in the general state of the system.
A model of the two entites and their interaction is available here.

Login feature

Since only parking managers should be able to perform tasks previously described, an authentication form must be deployed into the application. This brings the attention on how ParkServiceStatusGUI web page should be protected from undesiderd users and how to properly store managers credentials. The first point can be accomplished by easily deploy a login page which must redirects unauthorized users which try to access ParkServiceStatusGUI. The deployment can be faster achieved by using spring-boot-starter-security which meets spring boot framework already used to build ParkServiceGUI. At the same time managers' credentials should be stored in a dislocated server and crypted with any kind of hash function such as SHA-256 or MD5 to provent dictionary or injection attacks. However, for the sake of semplicity we can consider the deploy server for the ParkServiceStatusGUI safe and store credentials clearly in a file.

Project

Starting from what stated in the analysis phase, we introduce a new entity called managerService, taking part of the PMS which processes managers' requests.
The ParkServiceManagerGUI takes part of the MVC architecture developed in the Sprint 3. This pattern requires to split the ParkServiceManagerGUI into two components (Model not needed in our case):
  • View: user interfaces.
  • Controller: receives requests and generates content view for the manager and manage communication between the manager and the PMS.

In addition, parkingState resource has been updated with a new information that encodes an high temperature situation measured in the park.

managerService


In order to accept, queue and process different requests in a non-blocking manner, the managerService is implemented as an actor which has multiple status to exploit different tasks. managerService logic is message-based so that its state changes are triggered by message or events.
The managerService costantly provides information about current park status to the manager and manages trolley requests while the fan is managed automatically by the PMS, in particular by the ThermometerAdapter. The scheme on the side shows managerService behaviour.

managerService communication

managerService interacts with GuiAdapter, developed in the previous Sprint and updated with new features described below, from which receives manager's requests and to which sends tasks outcomes using a request/response logic.
In order to send QAK understandable messages, GuiAdapter uses connQakTcp.kt and MsgUtil.kt.

ParkmanagerServiceGUI

In order to get the current state of the park, we used a polling strategy which makes request every 2 seconds. It has been choosen since the manager GUI requires frequent live updates and polling fits well since it is not involved any real-time interval constraint.
Meanwhile the alarm related to the outdoor area and the high temperature notification requires the server pushing to the client. In order to achieve this architectural paradigm, communication is based over websockets using the STOMP messagging protocol included in the Spring Boot framework as explained here. Websocket avoids traffic overhead and it is used since those events might occur rarely.

GuiAdapter

The GuiAdapter developed in the last Sprint has been expanded with new methods to fulfill manager's requests.
Moreover GuiAdapter aim is to open the connection via websocket and add new callback functions to the thermometer and outdoor adapter which are called when their events occur.

ParkmanagerServiceView

Two html files have been written to represents client views:
  • login.html
  • homeMgmt.html
Together with this pages, mgmt.js is included to manage commincation with the server. They also dynamically modify html's DOM.
In order to remove situations of error related to trolley management, commands to change trolley state are available only in specific scenarios. In particular command to stop the trolley is available only if temperature is over TMAX and trolley is not stopped, and command to resume the trolley in available only if temperature is under TMAX and trolley is stopped.
As value for TMAX is chosen 30 degree celsius.

In addition in homeMgmt.html page is included the parking slots status, represented by a table with cells and green and red button, as shown in the image.

Deployment

This is the final Sprint and now we present the deployment of the full system. The application is made of three main components:
  • The virtual enviroment WEnv represented by a Docker image.
  • The ParkmanagerService which deals with business logic represented by a Docker image as well. This Docker image has been deployed starting from the image in the Sprint 3.
  • The basicRobot developed in Sprint 2 represented by a zip file.
The first two components have been deployed in a machine standing in the parking area meanwhile the third one has been deployed in the real DDR robot built by us.
In order to make system inizialization easier, a docker-compose file which includes WEnv and the PMS has been developed and deployed.

In order to make the system more flexible, in future we suggest to externalize some priviously embedded variables used by the system for the initial configuration.
